## 1. Overview

This sprint establishes the entire data persistence layer for Ippo. It has no dependencies of its own but every subsequent sprint — Auth, Assessment, Learning Path, Progress Tracking, Gamification, and Quizzes — depends on the schema defined here.

**What this sprint delivers:**

- Four production-ready Postgres tables with all constraints and indexes
- Drizzle ORM TypeScript schema (`/src/db/schema.ts`)
- Zod validation schemas for all API inputs that touch the DB
- Server route stubs (file structure + types) for TanStack Start
- Drizzle migration files in `/drizzle/`
- Database connection setup for Neon Postgres

**Why it is the foundation:**

Auth (Sprint 02) needs the `users` table to store accounts. Assessment (Sprint 03) stores results in `quiz_results`. Learning Path (Sprint 04) reads `progress` records. Gamification (Sprint 06) writes to `achievements`. Every sprint reads or writes to at least one table created here. Doing this right the first time — correct constraints, correct indexes, forward-compatible slug design — prevents costly migrations later.

## 3. Database Tables

### 3.1 `users`

Stores registered user accounts. Created on successful OTP verification via Better Auth.

```sql
CREATE TABLE users (
  id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  email            VARCHAR(255) UNIQUE NOT NULL,
  name             VARCHAR(255),
  preferred_language VARCHAR(2) DEFAULT 'id',     -- 'id' = Indonesian, 'en' = English
  assessed_level   VARCHAR(10),                   -- 'kana', 'n5', 'n4', 'n3', 'n2', 'n1'
  created_at       TIMESTAMP    DEFAULT now(),
  updated_at       TIMESTAMP    DEFAULT now()
);
```

**Field notes:**
- `id`: UUID v4 generated by Postgres (`gen_random_uuid()`). Never auto-increment integers; UUIDs are safe for distributed systems and prevent enumeration attacks.
- `email`: Unique, not null. The primary identifier for all auth flows.
- `preferred_language`: 2-char ISO code. Defaults to `'id'` (Indonesian). Used to serve Indonesian-first copy.
- `assessed_level`: Set when user completes assessment. Nullable — users who skip assessment have `NULL` here.

### 3.2 `progress`

Tracks completion of individual learning steps per user per level. One row per (user, level, step_slug) triple.

```sql
CREATE TABLE progress (
  id           UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  level        VARCHAR(10)  NOT NULL,    -- 'kana', 'n5', 'n4', 'n3', 'n2', 'n1'
  step_slug    VARCHAR(255) NOT NULL,    -- e.g. 'hiragana-gojuuon'
  completed    BOOLEAN      DEFAULT false,
  completed_at TIMESTAMP,               -- NULL until step is marked complete
  created_at   TIMESTAMP    DEFAULT now(),
  updated_at   TIMESTAMP    DEFAULT now()
);
```

**Field notes:**
- `user_id`: Foreign key with `ON DELETE CASCADE` — when a user account is deleted, all their progress rows are automatically removed.
- `level`: Denormalized for query performance. Indexed alongside `user_id` for fast per-level lookups.
- `step_slug`: Kebab-case identifier matching the TypeScript content files (e.g., `hiragana-gojuuon`). Must match exactly what is stored in localStorage for the guest → account migration to work correctly.
- `completed_at`: Only set when `completed = true`. Allows tracking when exactly a step was finished for streak calculations.

### 3.3 `quiz_results`

Records every quiz attempt (including the onboarding assessment). Used for score history, spaced repetition (future), and analytics.

```sql
CREATE TABLE quiz_results (
  id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  quiz_slug       VARCHAR(255) NOT NULL,    -- e.g. 'onboarding_assessment', 'n5-vocab-quiz-1'
  level           VARCHAR(10),              -- NULL for assessment; set for practice quizzes
  score           INTEGER      NOT NULL,
  total_questions INTEGER      NOT NULL,
  completed_at    TIMESTAMP    DEFAULT now(),
  created_at      TIMESTAMP    DEFAULT now()
);
```

**Field notes:**
- `quiz_slug`: Identifies the quiz type. The onboarding assessment uses `'onboarding_assessment'`. Practice quizzes (Sprint 08) will use slugs like `'n5-vocab-quiz-1'`.
- `level`: `NULL` for the onboarding assessment (level is the output, not the input). Set to the appropriate JLPT level for practice quizzes.
- `score` / `total_questions`: Stored as integers. Percentage is always computed at query time (`score * 100.0 / total_questions`), never stored.
- No `updated_at`: Quiz results are immutable records. A new row is inserted for each attempt; nothing is updated.

### 3.4 `achievements`

Records achievement unlocks. One row per (user, achievement_slug) — each achievement can only be unlocked once per user.

```sql
CREATE TABLE achievements (
  id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  achievement_slug VARCHAR(255) NOT NULL,    -- e.g. 'kana-master', 'first-step', 'streak-7'
  unlocked_at      TIMESTAMP    DEFAULT now(),
  created_at       TIMESTAMP    DEFAULT now()
);
```

**Field notes:**
- `achievement_slug`: Kebab-case identifier defined in the application's achievements registry (to be built in Sprint 06). Examples: `'sound-learner'` (first step completed), `'kana-master'` (all kana steps done), `'streak-7'` (7-day streak).
- `unlocked_at`: Separate from `created_at` for clarity and future flexibility (e.g., retroactively granting achievements).
- No `updated_at`: Like quiz results, achievements are immutable records.

## 4. Indexes

All indexes must be created in the same migration as their table.

```sql
-- Users: fast email lookup during auth
CREATE INDEX idx_users_email ON users(email);

-- Progress: fast lookup of all user steps within a level
CREATE INDEX idx_progress_user_level ON progress(user_id, level);

-- Progress: enforce one row per (user, level, step) combination
-- Used with ON CONFLICT for UPSERT pattern
CREATE UNIQUE INDEX idx_progress_unique ON progress(user_id, level, step_slug);

-- Quiz results: fast lookup ordered by most recent (for history + streak)
CREATE INDEX idx_quiz_results_user ON quiz_results(user_id, completed_at DESC);

-- Achievements: enforce one unlock per achievement per user
CREATE UNIQUE INDEX idx_achievements_unique ON achievements(user_id, achievement_slug);
```

**Index rationale:**

| Index | Why |
|---|---|
| `idx_users_email` | Auth flow does `SELECT * FROM users WHERE email = ?` on every login/register |
| `idx_progress_user_level` | Learning path renders `SELECT ... WHERE user_id = ? AND level = ?` on every page load |
| `idx_progress_unique` | Prevents duplicate progress rows; enables `ON CONFLICT DO UPDATE` UPSERT |
| `idx_quiz_results_user` | Quiz history and streak tracking need ordered results by user + date |
| `idx_achievements_unique` | Prevents duplicate achievement unlocks; enables idempotent insert |

## 5. Drizzle ORM Schema

Full TypeScript schema file. Create at `/src/db/schema.ts`.

```typescript
import {
  pgTable,
  uuid,
  varchar,
  boolean,
  timestamp,
  integer,
  index,
  unique,
} from 'drizzle-orm/pg-core';

// ─── Users ───────────────────────────────────────────────────────────────────

export const users = pgTable('users', {
  id:                uuid('id').primaryKey().defaultRandom(),
  email:             varchar('email', { length: 255 }).unique().notNull(),
  name:              varchar('name', { length: 255 }),
  preferred_language: varchar('preferred_language', { length: 2 }).default('id'),
  assessed_level:    varchar('assessed_level', { length: 10 }),
  created_at:        timestamp('created_at').defaultNow(),
  updated_at:        timestamp('updated_at').defaultNow(),
});

// ─── Progress ─────────────────────────────────────────────────────────────────

export const progress = pgTable(
  'progress',
  {
    id:           uuid('id').primaryKey().defaultRandom(),
    user_id:      uuid('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
    level:        varchar('level', { length: 10 }).notNull(),
    step_slug:    varchar('step_slug', { length: 255 }).notNull(),
    completed:    boolean('completed').default(false),
    completed_at: timestamp('completed_at'),
    created_at:   timestamp('created_at').defaultNow(),
    updated_at:   timestamp('updated_at').defaultNow(),
  },
  (table) => [
    index('idx_progress_user_level').on(table.user_id, table.level),
    unique('idx_progress_unique').on(table.user_id, table.level, table.step_slug),
  ],
);

// ─── Quiz Results ─────────────────────────────────────────────────────────────

export const quiz_results = pgTable(
  'quiz_results',
  {
    id:              uuid('id').primaryKey().defaultRandom(),
    user_id:         uuid('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
    quiz_slug:       varchar('quiz_slug', { length: 255 }).notNull(),
    level:           varchar('level', { length: 10 }),  // nullable
    score:           integer('score').notNull(),
    total_questions: integer('total_questions').notNull(),
    completed_at:    timestamp('completed_at').defaultNow(),
    created_at:      timestamp('created_at').defaultNow(),
  },
  (table) => [
    index('idx_quiz_results_user').on(table.user_id, table.completed_at),
  ],
);

// ─── Achievements ─────────────────────────────────────────────────────────────

export const achievements = pgTable(
  'achievements',
  {
    id:               uuid('id').primaryKey().defaultRandom(),
    user_id:          uuid('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
    achievement_slug: varchar('achievement_slug', { length: 255 }).notNull(),
    unlocked_at:      timestamp('unlocked_at').defaultNow(),
    created_at:       timestamp('created_at').defaultNow(),
  },
  (table) => [
    unique('idx_achievements_unique').on(table.user_id, table.achievement_slug),
  ],
);

// ─── Inferred Types ───────────────────────────────────────────────────────────

export type User            = typeof users.$inferSelect;
export type NewUser         = typeof users.$inferInsert;

export type Progress        = typeof progress.$inferSelect;
export type NewProgress     = typeof progress.$inferInsert;

export type QuizResult      = typeof quiz_results.$inferSelect;
export type NewQuizResult   = typeof quiz_results.$inferInsert;

export type Achievement     = typeof achievements.$inferSelect;
export type NewAchievement  = typeof achievements.$inferInsert;
```

## 9. Edge Cases & Constraint Handling

### 9.1 Unique Constraint Violations (Progress UPSERT)

The `idx_progress_unique` index on `(user_id, level, step_slug)` enforces one row per step. All writes to `progress` must use an UPSERT pattern, never a blind INSERT.

**Correct pattern (Sprint 05 implementation):**

```typescript
import { db } from '@/db';
import { progress } from '@/db/schema';
import { sql } from 'drizzle-orm';

async function markStepComplete(userId: string, level: string, stepSlug: string) {
  await db
    .insert(progress)
    .values({
      user_id:      userId,
      level,
      step_slug:    stepSlug,
      completed:    true,
      completed_at: new Date(),
    })
    .onConflictDoUpdate({
      target:  [progress.user_id, progress.level, progress.step_slug],
      set: {
        completed:    true,
        completed_at: new Date(),
        updated_at:   new Date(),
      },
    });
}
```

**Do not use:**
```typescript
// WRONG — will throw on second completion attempt
await db.insert(progress).values({ ... });
```

### 9.2 Cascade Deletes

`ON DELETE CASCADE` is set on `user_id` foreign keys in `progress`, `quiz_results`, and `achievements`. When a user account is deleted:

1. All `progress` rows for that user are deleted automatically
2. All `quiz_results` rows for that user are deleted automatically
3. All `achievements` rows for that user are deleted automatically

This behavior is intentional: GDPR compliance requires complete data removal on account deletion. **No additional cleanup code is needed in the application layer.**

### 9.3 UUID Generation

All `id` columns use `DEFAULT gen_random_uuid()` (Postgres built-in, requires `pgcrypto` extension on older Postgres). Neon Postgres has `pgcrypto` enabled by default. **Never generate UUIDs in application code** — let the database handle it.

In Drizzle ORM, this is `.defaultRandom()` which maps to `gen_random_uuid()`.

### 9.4 Timestamp Defaults

`created_at` and `updated_at` default to `now()`. In Drizzle, this is `.defaultNow()`.

**Important:** `updated_at` is NOT automatically updated by Postgres on row changes (unlike some ORMs). Any UPDATE query that modifies a row must also set `updated_at = now()` explicitly.

```typescript
// Always include updated_at in UPDATE queries
await db
  .update(progress)
  .set({ completed: true, updated_at: new Date() })
  .where(/* ... */);
```

### 9.5 Achievement Unique Constraint

`idx_achievements_unique` on `(user_id, achievement_slug)` prevents the same achievement from being unlocked twice. Use the same UPSERT-or-ignore pattern:

```typescript
await db
  .insert(achievements)
  .values({ user_id: userId, achievement_slug: slug })
  .onConflictDoNothing();  // Silently skip if already unlocked
```

## 10. Migration Strategy

### 10.1 Directory Structure

```

├── 0000_initial_schema.sql     (all 4 tables + all indexes)
├── meta/
│   └── _journal.json           (Drizzle migration metadata)
```

### 10.2 Generating Migrations

After any schema change in `/src/db/schema.ts`:

```bash
npx drizzle-kit generate
```

This creates a new `.sql` file in `/drizzle/`. Commit this file alongside the schema change.

### 10.3 Applying Migrations

**Before every production deploy:**

```bash
npx drizzle-kit migrate
```

This runs all pending migrations against the target database (set via `DATABASE_URL`).

For Vercel, add this to the build command or as a pre-deploy hook:

```bash
npx drizzle-kit migrate && npm run build
```

### 10.4 Never Edit Applied Migrations

Once a migration file has been applied to production, it must never be edited. If a mistake is found, create a new migration that corrects it.

### 10.5 Future Slug Change Compatibility

From PRD Section 11.2: if a step slug is renamed in the future (e.g., `hiragana-gojuuon` → `hiragana-basic`), backward compatibility is handled via:

**Option A — Migration script (preferred):**
```sql
UPDATE progress SET step_slug = 'hiragana-basic' WHERE step_slug = 'hiragana-gojuuon';
```

**Option B — View alias (if migration is too risky):**
```sql
-- Application queries this view instead of the table directly
CREATE VIEW progress_with_aliases AS
  SELECT *, 
    CASE step_slug
      WHEN 'hiragana-gojuuon' THEN 'hiragana-basic'
      ELSE step_slug
    END AS canonical_step_slug
  FROM progress;
```

**Rule:** Any content file change that renames a `step.slug` must be accompanied by a database migration that updates all existing `progress` rows referencing the old slug. This prevents ghost progress records (rows that no longer match any known step).

## 11. Performance Targets

| Query | Target | Index Used |
|---|---|---|
| `SELECT * FROM users WHERE email = ?` | <50ms | `idx_users_email` |
| `SELECT * FROM progress WHERE user_id = ? AND level = ?` | <100ms | `idx_progress_user_level` |
| `INSERT INTO progress ON CONFLICT DO UPDATE` | <150ms | `idx_progress_unique` |
| `SELECT * FROM quiz_results WHERE user_id = ? ORDER BY completed_at DESC` | <100ms | `idx_quiz_results_user` |
| `SELECT * FROM achievements WHERE user_id = ?` | <100ms | `user_id` (covered by unique index) |

**All queries must target <500ms end-to-end** including network latency to Neon (typically 20–80ms from Vercel Edge).

### 11.1 Query Optimization Rules

- Always filter by `user_id` first — it is the leading column in all composite indexes
- Never `SELECT *` in production code — select only needed columns
- For the Learning Path page, fetch progress for a single level at a time (not all levels at once)
- Use TanStack Query on the client with `staleTime: 1000 * 60 * 5` (5 minutes) to avoid redundant DB calls
